* TCP 问题详解

影响 TCP 性能的主要因素有三个 (还有其他因素, 但这三个是核心):
- packet loss
- 延迟 (即 RTT —— 往返时间)
- buffer/window size


** 延迟/RTT

这是一台主机从另一台主机获取已发送信息所需的最短时间

这点通常没什么可说的, 由服务端到客户端的距离决定

延迟本身也会很大程度上影响其它两个因素


** buffers/windows size

*** 名词解释

- *内核缓冲区*: 操作系统内核为 TCP 连接分配的内存, 用来暂存发送或接收的数据。
- *发送缓冲区 / send buffer*: 发送端内核中存放应用已写入、但尚未完全确认的数据的地方。
- *接收缓冲区 / receive buffer*: 接收端内核中存放已经收到、但应用还没读取的数据的地方。
- *TCP window*: TCP 允许对端继续发送的数据量限制。
- *接收窗口 / receive window / rwnd*: 接收方告诉发送方"我还能接收多少数据"。
- *通告窗口 / advertised window*: 接收方通过 TCP 报文头告诉发送方的接收窗口大小。
- *拥塞窗口 / congestion window / cwnd*: 发送方根据网络拥塞情况自己计算出的发送上限。
- *实际发送窗口*: 发送方当前最多能发送但尚未确认的数据量, 通常约等于 min(rwnd, cwnd)。
- *in-flight data / 在途数据*: 已经发送出去、但还没有收到 ACK 确认的数据。
- *ACK*: 接收方发给发送方的确认消息, 表示某些数据已经收到。
- *RTT / 往返时延*: 一个数据包从发送方到接收方, 再等确认返回所需的时间。
- *带宽 / bandwidth*: 链路单位时间内理论上能传输的数据量。
- *吞吐量 / throughput*: 实际单位时间内成功传输的数据量。
- *BDP / 带宽时延积*: 跑满链路所需的在途数据量, 计算方式是 带宽 × RTT。
- *window 小于 BDP*: TCP 允许在途的数据太少, 无法填满链路。
- *发满窗口后等待*: 发送方把允许发送的数据发完后, 必须等 ACK 返回才能继续发送。
- *TCP buffer autotuning*: 内核自动调整 TCP 缓冲区和窗口大小的机制。
- *window scaling*: TCP 扩大窗口字段表达范围的机制, 使窗口可以大于 65,535 字节。
- *zero window*: 接收方通告窗口为 0, 表示接收缓冲区满了, 发送方暂时不能继续发送。
- *packet loss / 丢包*: 数据包在网络中丢失, TCP 通常需要重传。
- *retransmission / 重传*: 发送方重新发送之前未被确认的数据。
- *MTU*: 网络链路单个包可承载的最大传输单元。
- *PMTU*: 从发送方到接收方整条路径上允许通过的最大包大小。
- *PMTU blackhole*: 大包过不去, 相关 ICMP 又被阻断, 导致连接卡顿或吞吐极低。
- *bufferbloat*: 缓冲区过大导致数据长时间排队, 延迟升高。
- *应用层瓶颈*: 速度慢的原因在应用程序、磁盘、数据库、协议设计等, 而不是网络本身。
- *网络层瓶颈*: 速度慢的原因在链路、路由、丢包、限速、MTU、拥塞等网络因素。


*** 原理

window 越大, 两台主机之间可传输的数据就越多。

需要注意的是, 如果窗口小于可用带宽乘以延迟, 即带宽延迟积 (BDP, Bandwidth Delay Product, BDP = 带宽 x RTT), 发送方将发送一个完整窗口的数据, 然后等待接收方确认数据。这会导致性能下降, 也是 TCP buffer autotuning 存在的主要原因

BDP 表示: 要把这条链路"填满", 网络中同时需要有多少数据在飞

发送方不能只发一点点数据就等 ACK; 它必须允许足够多的数据同时"在路上"

没有 autotuning 时, 你需要手动设置系统全局参数, 针对每个目的地进行优化, 例如 Linux 的:

#+begin_src
net.ipv4.tcp_rmem
net.ipv4.tcp_wmem
net.core.rmem_max
net.core.wmem_max
#+end_src

有了 autotuning 后, 内核会根据每个链接的延迟, 自动调整 TCP buffers/windows 大小

这很好, 这比过去好多了, 只要网络良好, 如今的性能通常都相当不错


** packet loss

开始变得复杂起来喽

TCP 遇到 packet loss 时, 会
- 降低发送速率, 实现机制是缩小发送方的 window
- 然后会逐步恢复发送速率, 希望这次丢包只是暂时的

与过去多年相比, 现在的情况已经好多了……像 htcp 和 cubic 这类算法在恢复速率方面比旧方案 (TCP Reno) 要激进得多

TCP 会随着时间, 从较小的 window 开始逐步扩大 window, 延迟越长, 完成这个控制循环所需的时间就越长。因此, RTT 越长, 恢复的时间就越长

但是, 当路由器、交换机和防火墙中增加小缓冲区时, 还会出现其他问题, 使丢包与延迟之间的相互作用变得更加复杂

当延迟较大时 (超过约 10 ms 就开始成为问题, 而 20 ms 以上则变得非常棘手), 需要较大的窗口才能实现足够的吞吐量。当窗口较大时, TCP 可以一次性发送大量数据, 网卡通常不了解也不关心 TCP 协议, 它只知道当有待发送的数据包时, 就会将这些数据包以配置的链路速度 (例如 10Gbps) 连续发送到链路上。

因此, 如果 TCP 窗口为 4MB, TCP 会将 4MB 数据传递给网卡, 而网卡会以 10Gbps 的速度将其全部发送到链路上, 这意味着发送方与接收方路径上的每个设备都会遭遇高速数据包突发, 如果路径上任何位置存在缓冲或排队问题, 部分数据包就会被丢弃, 从而导致 TCP 降低发送速率。

如果在低延迟环境下观察, 通常不会察觉到这个问题; 若出现零星丢包现象, 此时低延迟 TCP 能快速恢复, 用户通常不会感知到性能损失

即使路径中存在小型交换机 buffers, 它们通常不会导致低延迟连接丢包, 因为 TCP 发送的数据突发量不足以引发丢包

但是, 如果是交换机小 buffer + 高延迟环境下, 就会观察到显著的性能下降。在随机错误场景下, TCP 将始终处于恢复模式, 窗口持续保持较小状态。而在小型缓冲区场景下, TCP 会经历 "窗口增长→遭遇丢包→窗口缩减→再次增长" 的循环, 导致窗口长期处于较小状态, 性能表现不佳


** Packet reordering / 包乱序

*** 名词解释

- *Packet reordering / 包乱序*: 后发的包先到, 先发的包后到。
- *OOP / Out-of-Order Packet*: 乱序到达的包。
- *IOP / In-Order Packet*: 按正常顺序到达的包。
- *jitter / 抖动*: 包到达时间不稳定。
- *SACK*: TCP 用来告诉发送方"中间有些包没收到, 但后面的包收到了"的机制。
