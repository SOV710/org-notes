#+title: Pacet Ordering / Reordering
#+author: SOV710
#+date: 2026-05-17 14:02:46
#+description: 传输控制协议 (TCP) 对端到端发送的数据包行为非常敏感。到达时间的抖动 ("jitter") 以及可能改变数据段顺序或完全丢失数据段的其他异常事件, 会引发难以恢复的问题


* 名词解释

- *Packet reordering / 包乱序*: 后发的包先到, 先发的包后到。
- *OOP / Out-of-Order Packet*: 乱序到达的包。
- *IOP / In-Order Packet*: 按正常顺序到达的包。
- *jitter / 抖动*: 包到达时间不稳定。
- *SACK*: TCP 用来告诉发送方"中间有些包没收到, 但后面的包收到了"的机制。


* 关键结论

TCP 看到乱序时, 可能误以为丢包了, IOP 和 OOP 之间的时间间隔增大时, 会触发 SACK 行为, 这是一种预防措施, 但会在高延迟环境下, 会导致窗口阻塞

包只是晚到, 不一定真丢了, 但 TCP 接收方会发现序号中间缺了一段, 于是触发 SACK / 重传相关行为

偶尔乱序还能恢复; 频繁乱序会让 TCP 持续误判、重传、缩小有效发送能力

乱序包延迟越久, 问题越严重: 如果缺失的包很快补上, 影响较小; 如果晚到时间较长, TCP 更可能触发重传和窗口阻塞

观察表示, 当乱序延迟大约超过 RTT 的 1% 时, 进入明显恶化区间

调 TCP 参数只能缓解低水平乱序, 不能解决严重乱序

以下场景可能导致此类情况发生:
- 数据流经可能并行处理单个 TCP 流的设备 (如防火墙、数据包整形器)
- 数据流经可能将流量分配到等效链路的设备 (如主机上的捆绑链路、网络设备上的链路聚合组 LAG), 且哈希算法未对同一流中的所有数据包使用相同路径时
- 由于单个流中的部分流量以不同优先级转发, 导致路径中某处发生排队
- 自适应负载均衡方法 (例如在主机或路由器内部)

采用 Science DMZ 架构有助于消除此问题
