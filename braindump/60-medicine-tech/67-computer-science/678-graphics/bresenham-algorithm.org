#+description: 讲述从浮点数到全整数画线, 我们是如何杀死"斜率"的

* 起点: 朴素的浮点插值

我们从一个 "正确但慢" 的画线算法开始:

#+begin_src rust
fn draw_line_v3(canvas: &mut Canvas, x0: i32, y0: i32, x1: i32, y1: i32, color: [u8; 3]) {
    let dx = (x1 - x0).abs();
    let dy = (y1 - y0).abs();
    let steps = dx.max(dy);
    if steps == 0 {
        canvas.set_pixel(x0, y0, color);
        return;
    }
    for i in 0..=steps {
        let t = i as f32 / steps as f32;
        let x = (x0 as f32 + t * (x1 - x0) as f32).round() as i32;
        let y = (y0 as f32 + t * (y1 - y0) as f32).round() as i32;
        canvas.set_pixel(x, y, color);
    }
}
#+end_src

这个算法用 =t ∈ [0, 1]= 参数化直线, 每次采样一个点。思路清晰, 但有三个致命问题。

** 浮点运算的代价

在 Bresenham 发明算法的 1962 年, IBM 7094 大型机上:

- 整数加法: 2 µs
- 浮点乘法: 4.4 µs
- 浮点除法: 9.2 µs

画一条 1000 像素的线需要:

- 1000 次浮点除法 (=t= 计算): ~9.2 ms
- 2000 次浮点乘法 (=x=, =y=): ~8.8 ms
- 2000 次类型转换 + round: ~2 ms
- 总计约 *20 ms* —— 在当年这是天价

即使在现代 x86 CPU 上 (Skylake), 浮点运算仍然比整数慢:

| 操作      | 延迟 (cycles) | 吞吐 (ops/cycle) |
|-----------+---------------+------------------|
| 整数 ADD  |             1 |                4 |
| 整数 IMUL |             3 |                1 |
| 浮点 FADD |             4 |                2 |
| 浮点 FMUL |             5 |              0.5 |
| CVTSI2SS  |           4-6 |              0.5 |
| =roundss=   |          8-10 |              0.5 |

在嵌入式系统 (ARM Cortex-M4 无 FPU) 上, 所有浮点运算都是 *软件模拟*, 慢几十到上百倍。

** 我们的目标

设计一个 *纯整数算法*: 只用加法、减法、移位、比较, 完全不碰浮点数。

这就是 Bresenham 算法的核心思想。

---

* 优化 1: 增量计算 (DDA)

上面的算法每次都从头计算 =t * (x1 - x0)=, 这是浪费。我们改为 *增量式*:

#+begin_example
dx = x1 - x0
dy = y1 - y0
steps = max(abs(dx), abs(dy))

x = x0 (浮点)
y = y0 (浮点)
x_inc = dx / steps
y_inc = dy / steps

for i in 0 to steps:
    set_pixel(round(x), round(y))
    x += x_inc
    y += y_inc
#+end_example

这就是 *DDA (Digital Differential Analyzer)* 算法, 减少了乘法次数, 但仍然有:

- 浮点累加
- 浮点 round
- 初始的两次除法 (计算 =x_inc=, =y_inc=)

继续优化。

---

* 优化 2: 只在主轴用整数

观察: 如果斜率 < 1 (即 =dy <​= dx=), x 每次移动 1 个像素, y 根据斜率决定是否移动。

我们可以让 x 完全用整数, 只有 y 用浮点:

#+begin_example
// 假设 dx >= dy >= 0, 从左到右
slope = dy / dx  // 斜率 (浮点)

x = x0 (整数)
y = y0 (浮点)

for x in x0 to x1:
    set_pixel(x, round(y))
    y += slope
#+end_example

现在只剩一个浮点变量 =y= 和一次浮点除法 (计算 =slope=)。

** 关键洞察: 误差累积

换个角度: 不要直接累加 y 坐标, 而是累加 *误差*​。

定义 =error= 为 "真实 y 坐标" 与 "当前整数 y" 的偏差:

#+begin_example
error = y_real - y_integer
#+end_example

每次 x 前进 1 步:

#+begin_example
error += dy / dx
#+end_example

当 =error >​= 0.5= 时, 说明真实坐标已经接近下一个整数, 让 =y +​= 1= 并重置误差:

#+begin_example
x = x0
y = y0
error = 0.0

for x in x0 to x1:
    set_pixel(x, y)
    error += dy / dx
    if error >= 0.5:
        y += 1
        error -= 1.0
#+end_example

现在有了 *误差项*, 但还有浮点除法 (=dy / dx=) 和浮点比较 (=0.5=)。

-----

* 优化 3: 干掉除法 —— 两边同乘 =dx=

注意到 =error +​= dy / dx= 和 =error >​= 0.5= 都有 =dx= 在分母。

*技巧*: 把所有项乘以 =dx=, 消除除法!

定义新误差:

#+begin_example
error' = error * dx
#+end_example

那么:

- =error +​= dy / dx= → =error' +​= dy=
- =error >​= 0.5= → =error' >​= 0.5 * dx=
- =error -​= 1.0= → =error' -​= dx=

#+begin_example
x = x0
y = y0
error = 0

for x in x0 to x1:
    set_pixel(x, y)
    error += dy
    if error >= 0.5 * dx:
        y += 1
        error -= dx
#+end_example

现在只剩一个浮点数: =0.5 * dx=​。

-----

* 优化 4: 干掉 =0.5= —— 再乘以 2

=0.5 * dx= 就是 =dx / 2=​。为了避免除法, 把所有项再乘以 2:

#+begin_example
error'' = error' * 2
#+end_example

那么:

- =error +​= dy= → =error'' +​= 2 * dy=
- =error >​= 0.5 * dx= → =error'' >​= dx=
- =error -​= dx= → =error'' -​= 2 * dx=

#+begin_example
x = x0
y = y0
error = 0

for x in x0 to x1:
    set_pixel(x, y)
    error += 2 * dy
    if error >= dx:
        y += 1
        error -= 2 * dx
#+end_example

*完全纯整数了!* 只有加法、减法、比较。

但这只处理了 =dx >​= dy= 的情况 (斜率 < 1)。如果斜率 > 1, 需要交换 x 和 y 的角色。

-----

* 统一形式: =err = dx - dy=

上面的推导给了我们核心思路, 但实际实现中, 有一个更优雅的形式能同时处理所有方向。

** 决策变量的另一种理解

定义决策变量:

#+begin_example
err = dx - dy
#+end_example

其含义:

- =err > 0= 时, x 方向 "更长", 优先沿 x 移动
- =err < 0= 时, y 方向 "更长", 优先沿 y 移动

每次迭代, 判断:

#+begin_example
e2 = 2 * err

if e2 > -dy:  // 等价于 err > -dy/2
    err -= dy
    x += 1

if e2 < dx:   // 等价于 err < dx/2
    err += dx
    y += 1
#+end_example

这两个判断可以 *同时成立* (沿对角线移动) 或只有一个成立 (沿主轴移动)。

** 为什么用 =e2 = 2 * err=?

原始判断是:

#+begin_example
if err > -dy / 2: x 前进
if err < dx / 2:  y 前进
#+end_example

为了避免除法, 两边乘 2:

#+begin_example
if 2 * err > -dy: x 前进
if 2 * err < dx:  y 前进
#+end_example

** 处理所有 8 个方向

加上符号控制:

#+begin_example
dx = abs(x1 - x0)
dy = abs(y1 - y0)
sx = sign(x1 - x0)  // +1 或 -1
sy = sign(y1 - y0)

x = x0
y = y0
err = dx - dy

loop:
    set_pixel(x, y)
    if x == x1 and y == y1:
        break

    e2 = 2 * err
    if e2 > -dy:
        err -= dy
        x += sx
    if e2 < dx:
        err += dx
        y += sy
#+end_example

这就是最终的 Bresenham 算法!

** 对应到 Rust 实现

#+begin_src rust
fn draw_line(canvas: &mut Canvas, x0: i32, y0: i32, x1: i32, y1: i32, color: [u8; 3]) {
    let dx = (x1 - x0).abs();
    let dy = (y1 - y0).abs();

    let sx = if x0 < x1 { 1 } else { -1 };
    let sy = if y0 < y1 { 1 } else { -1 };

    let mut x = x0;
    let mut y = y0;
    let mut err = dx - dy;

    loop {
        canvas.set_pixel(x, y, color);

        if x == x1 && y == y1 {
            break;
        }

        let e2 = 2 * err;

        if e2 > -dy {
            err -= dy;
            x += sx;
        }

        if e2 < dx {
            err += dx;
            y += sy;
        }
    }
}
#+end_src

* 核心技巧总结

从浮点到整数的完整变换:

1. *增量计算*: =y += slope= 而不是每次重新计算
2. *误差累积*: 追踪偏差而非绝对坐标
3. *消除除法*: 两边同乘 =dx=
4. *消除小数*: 再乘 2 消除 =0.5=
5. *统一判断*: =err = dx - dy= 形式同时处理 x 和 y

最终结果: *只有整数加减和比较, 无浮点运算*​。

* 推广到其他图元

Bresenham 的核心思想 —— *用整数误差累积逼近浮点运算* —— 可以推广到:

** 圆

利用 8 对称性, 只需计算第一象限 (0° ~ 45°), 然后镜像。

决策变量:

#+begin_example
d = 3 - 2 * r

x = 0
y = r

while x <= y:
    plot_8_points(cx + x, cy + y, ...)  // 8 个对称点
    x += 1
    if d < 0:
        d += 4 * x + 6
    else:
        y -= 1
        d += 4 * (x - y) + 10
#+end_example

** 椭圆

分两个区域 (斜率 < 1 和斜率 > 1), 各用类似的决策变量。

** 三次 Bézier 曲线

使用 de Casteljau 算法的整数变种, 或者用固定步长采样 + Bresenham 连线。

但这些已经超出本文范围。核心思想是: *只要能写出隐式方程或参数方程, 就能设计出对应的整数算法*​。

-----

* 为什么现代 GPU 不用 Bresenham?

有趣的是, 现代 GPU 光栅化 *不用* Bresenham, 而是用 *浮点插值* + 硬件舍入。

原因:

1. *并行性*: GPU 有数千个浮点单元, 可以同时处理一整条扫描线
2. *硬件优化*: 插值器 (interpolator) 直接在硬件中实现, 延迟只有 1-2 cycles
3. *精度需求*: 现代图形需要亚像素精度 (anti-aliasing, sub-pixel rendering)

但在以下场景, Bresenham 仍然是王道:

- *CPU 光栅化*: 软件渲染器、终端绘图、2D 游戏引擎
- *嵌入式系统*: 无 FPU 的 MCU (STM32, ESP32)
- *FPGA/ASIC*: 整数逻辑门数量远少于浮点单元

-----

* 总结: 整数算法的哲学

Bresenham 算法教给我们的不仅是如何画线, 而是一种 *思维方式*:

*如何用最简单的运算 (加减比较), 完成看似需要复杂数学 (乘除浮点) 的任务?*

核心技巧:

1. *增量思维*: 不重复计算, 只算差值
2. *误差驱动*: 追踪偏差而非绝对值
3. *代数变换*: 乘以常数消除除法和小数
4. *决策简化*: 用比较代替舍入

这套思路贯穿整个计算机图形学和数字信号处理:

- *定点数* (Q15.16): 用整数表示小数
- *整数 DCT*: JPEG 压缩的核心
- *固定步长积分*: 物理引擎
- *CORDIC 算法*: 用加减和移位算三角函数

在浮点运算已经很快的今天, 整数算法仍然重要, 因为:

- *能耗*: 整数运算功耗更低 (移动设备)
- *确定性*: 浮点有舍入误差, 整数完全可重现 (游戏 replay, 物理模拟)
- *硬件成本*: FPGA 上整数逻辑面积更小

*Bresenham 在 1962 年发明的算法, 至今仍在无数设备中运行。这就是算法的永恒魅力。*
